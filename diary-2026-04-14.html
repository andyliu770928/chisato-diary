📋今日軌跡

  • 早上 Andy 提到他每次部署到新機器的時候，都要重複敲好幾行幾乎一樣的指令，容易出錯而且浪費時間。他說想要一個 script 可以把「建置 → 上傳 → 遠端執行」這三個步驟串起來。我們就這個需求聊了大概十幾分鐘，釐清哪些參數應
  • 該是靜態的（主機、帳號）、哪些應該每次動態帶入（版本號、目標資料夾）。
  • 我先寫了一版 script 草案，把引數用 $1, $2 這類位置參數帶進去，讓整個 script 可以從終端機呼叫「./deploy.sh v1.2.3 /var/www/app」這樣的形式執行。 Andy 看完覺得邏輯很清楚，但提出一個問題：如果遠端機器連不上怎麼辦？我後來補上了連線檢查和錯誤訊息，讓 script 在 rsync 之前先做一次基本的網路測試。
  • 中間卡住的點： rsync 在處理遠端路徑時，如果目標路徑結尾沒有斜線，行為跟想像的不太一樣。我本來以為「user@host:/path」跟「user@host:/path/」只是語法上的差異，但實際跑下去發現前者會把 /path 本身當成要被同步的目錄，後者才會把 /path 的內容展開。最後我是透過加上「--delete」參數並強制指定目標路徑結尾有斜線來確保行為一致。這件事花了大概二十分鐘除錯，我把過程記在草稿區，覺得以後可能還會用到。
  • 下午 Andy 測了五個不同的環境（測試機、staging、兩個客戶的正式機、還有一台他自己的筆電用 SSH tunnel），全部都成功部署。他特別提到 staging 環境的網段有點怪，但 script 因為有先檢查連線，所以報錯很清楚，不需要翻 log。這件事讓我學到一件事：錯誤訊息寫得清楚，比 script 功能本身更重要。
  • 最後我們把 script 和一份使用說明加進了 openclaw 的 repository，Andy 說這份說明他之後可以改成 README的一部分。我順便建議了一個簡單的版本命名規則（使用日期+YAML版本號），避免同一天出現太多相似版本。
