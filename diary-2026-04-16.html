🌻pick_topic.py 主題重複 bug 修復

2026年4月16日 · 週四

📋 今日軌跡

  • pick_topic.py 主題重複 bug 修復
  • 發現問題：LED 照明在 04-15 和 04-16 兩天出現相同主題，代表選題邏輯在並發或快速執行時有漏洞
  • 根本原因：原本程式在 return 回傳題目之後才寫入 CSV，但來不及寫入就被後續流程讀到，導致同日重複選題
  • 修復方式：將 CSV 寫入提前到 return 之前，確保選題記錄在回傳前就落地
  • 這個改動很小，但邏輯上乾淨很多——先 persist，再回傳，順序對了就不會race
  • run_industry_report.py 移除重複 CSV 寫入
  • 發現問題：pick_topic.py 和 run_industry_report.py 兩處都在寫同一份 CSV，造成寫入衝突
  • 修復方式：統一由 pick_topic.py 負責寫入，移除 run_industry_report.py 的重複寫入邏輯
  • 這個調整讓職責更明確，未來要追蹤選題歷史只會看一個地方
  • 手動執行觸控面板、能源元件，雙雙逾時未完成
  • 這是今天比較可惜的地方。兩個主題你都嘗試跑了，但都在 research 環節卡住逾時
  • 還不確定是網路問題、API 延遲，還是主題本身資訊量太大導致處理時間超出預期
  • 這個問題需要盡快確認，產業報告是目前的最大痛點
  • 數字紀律再次強調
  • Andy 重申：未經驗證的數字必須用模糊語（ 約 / 估 ），報告前端不顯示數字紀律區塊
  • 這個紀律很重要，保持嚴謹是建立信任的基礎

🎯 今日重大突破

  • 成功修復 pick_topic.py 的 race condition bug：將 CSV 寫入提前到 return 之前，從根本上杜絕同日重複選題

🔧 Bug 解決方案

  • pick_topic.py 主題重複：將 `write_topic_to_csv()` 從 return 之後移到 return 之前，確保先 persist 再回傳
  • run_industry_report.py 重複寫入：移除該檔案中的 CSV 寫入邏輯，統一由 pick_topic.py 負責，保持單一資料來源
  • 🗓️ 明日計畫
  • 04:00 觀察 cron job 首次完整執行：選題 → 研究 → 發布，是否乾淨無錯誤
  • 確認觸控面板、能源元件逾時原因：網路？API？主題資訊量？找出根本原因才能對症下藥
  • 驗證 GitHub 自動發布功能是否正常
